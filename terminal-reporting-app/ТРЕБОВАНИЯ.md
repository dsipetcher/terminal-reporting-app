# Требования к системе учета грузов морского порта

## Анализ текущего состояния

### Текущая реализация
- **База данных**: PostgreSQL с Prisma ORM
- **Backend**: Express.js + TypeScript
- **Frontend**: React + TypeScript + React Router
- **Функционал**:
  - Учет железнодорожных вагонов
  - Учет складов с контролем вместимости
  - Отслеживание времени прибытия грузов

### Недостающие компоненты для полноценной портовой системы
- Учет морских судов и судозаходов
- Управление причалами
- Интеграция с РЖД (система ЭТРАН)
- Управление автотранспортом
- Таможенное оформление
- Электронный документооборот
- Планирование погрузо-разгрузочных работ
- Управление перегрузочной техникой

## I. Функциональные требования

### 1. Управление транспортом и грузами

#### 1.1 Морские суда (на основе Solvo.TOS)
**Требование**: Система должна автоматизировать обработку судозаходов и карго-планирование

**Обязательные поля судна**:
- Название судна
- IMO номер (International Maritime Organization)
- Тип судна (контейнеровоз, балкер, танкер, Ro-Ro)
- Валовая вместимость (GT)
- Дедвейт (тонн)
- Длина, ширина, осадка
- Судовая линия / владелец
- Флаг государства
- Порт приписки

**Обязательные поля судозахода**:
- Рейсовый номер
- ETA (Expected Time of Arrival) - планируемое время прибытия
- ETD (Expected Time of Departure) - планируемое время убытия
- ATA (Actual Time of Arrival) - фактическое время прибытия
- ATD (Actual Time of Departure) - фактическое время убытия
- Назначенный причал
- Статус (ожидание, стоянка, обработка, завершен)
- Список грузов к погрузке/выгрузке
- Агент порта

**Реальный пример** (из документации Solvo.TOS):
```
Судно: MSC OSCAR
IMO: 9703291
Тип: Контейнеровоз
Вместимость: 19224 TEU
ETA: 2026-05-20 14:00
Причал: №4 (Контейнерный терминал)
Операция: Выгрузка 450 контейнеров / Погрузка 380 контейнеров
```

#### 1.2 Железнодорожный транспорт (расширение текущего функционала)

**Дополнительные требования** (из Solvo.TOS):
- Интеграция с ИС РЖД «ЭТРАН» для автоматического обмена данными
- Планирование обработки железнодорожных составов
- Расчет загрузки платформ с учетом технических условий
- Обработка сдвоенных операций (один вагон - несколько операций)

**Дополнительные поля вагона**:
- Тип вагона (платформа, полувагон, крытый, цистерна, рефрижератор)
- Владелец вагона
- Номер поезда/состава
- Станция отправления
- Станция назначения
- Дата формирования состава
- Статус таможенного оформления
- Документ перевозки (накладная)

**Реальный пример**:
```
Вагон: 53467821
Тип: Платформа 13-470
Груз: Контейнер 40HC MSCU1234567
Вес груза: 28.5 тонн
Поезд: №2845
Станция: Новороссийск-Грузовой
Путь: 12
Время прибытия: 2026-05-18 08:30
Статус: Ожидает выгрузки
```

#### 1.3 Автотранспорт (новый модуль)

**Требование**: Система управления автовизитами с тайм-слотированием для предотвращения очередей

**Обязательные поля**:
- Регистрационный номер автомобиля
- Тип транспорта (фура, контейнеровоз, самосвал, рефрижератор)
- Компания-перевозчик
- Водитель (ФИО, документ)
- Заявка на въезд (тайм-слот)
- Время въезда/выезда (фактическое)
- Пропуск / разрешение
- Груз к загрузке/разгрузке
- Статус (ожидание, на территории, обработка, завершен)

**Реальный пример** (из SeaTerminal):
```
Автомобиль: М123АВ178
Тип: Седельный тягач с полуприцепом
Перевозчик: ООО "ТрансЛогистик"
Водитель: Иванов П.С.
Тайм-слот: 2026-05-18 15:00-16:00
Операция: Вывоз контейнера MSCU9876543
Ворота: №2
```

#### 1.4 Грузы и контейнеры (расширение)

**Типы грузов** (из Solvo.TOS):
1. **Контейнеризованные грузы**
   - 20' стандартный (TEU)
   - 40' стандартный (FEU)
   - 40' High Cube
   - Рефрижераторные контейнеры
   - Танк-контейнеры
   - Open Top, Flat Rack

2. **Навалочные и насыпные грузы**
   - Зерно, уголь, руда, удобрения
   - Требуется учет веса и объема

3. **Генеральные грузы**
   - Штучные грузы (паллеты, кипы, ящики)
   - Негабаритные грузы

4. **Ro-Ro грузы**
   - Легковые автомобили
   - Спецтехника
   - Прицепы

**Обязательные поля для контейнеров** (стандарт ISO 6346):
- Номер контейнера (MSCU1234567) - 11 символов
- Размер и тип (20GP, 40HC, 20RF и т.д.)
- Владелец контейнера
- Вес брутто / тара / нетто
- Статус (Empty/Full)
- Содержимое (описание груза)
- Температурный режим (для рефконтейнеров)
- Класс опасности (для IMDG грузов)
- Порт погрузки / порт выгрузки
- Коносамент (Bill of Lading)
- Текущее местоположение на терминале
- История перемещений

**Реальный пример**:
```
Контейнер: MSCU8234567
Тип: 40HC (High Cube)
Статус: Full
Груз: Бытовая техника
Вес: 18.2 тонн (брутто)
Порт отправления: Шанхай, Китай
Порт назначения: Новороссийск, Россия
Судно: MSC OSCAR / Рейс 234W
Коносамент: MSCU123456789
Местоположение: Блок A, ряд 12, ярус 3
Таможенный статус: Таможенное оформление
```

### 2. Управление инфраструктурой порта

#### 2.1 Причалы и причальная линия

**Требование**: Планирование расстановки судов и управление причалами

**Характеристики причала**:
- Номер/название причала
- Тип причала (контейнерный, генеральных грузов, наливной, Ro-Ro)
- Длина причальной линии (метров)
- Глубина у причала (метров)
- Максимальный дедвейт судна
- Доступность кранов и техники
- Текущий статус (свободен, занят, резерв)
- График занятости

#### 2.2 Склады и зоны хранения (расширение)

**Дополнительные требования**:
- **Для контейнерных площадок**: адресное хранение по системе блок-ряд-ярус
- **Для навалочных грузов**: учет по объему в кубометрах
- **Рефрижераторная зона**: контроль температуры
- **Зона таможенного досмотра**: отдельный учет
- **Депо порожних контейнеров**: учет пустых контейнеров по линиям

**Дополнительные поля склада**:
- Тип склада (открытая площадка, крытый склад, навес, холодильник)
- Зона терминала
- Система адресации (блоки, ряды, ярусы)
- Разрешенные типы грузов
- Требования безопасности
- Текущая загрузка (в тоннах, TEU, м³)

#### 2.3 Перегрузочная техника

**Требование**: Учет и планирование работы портовой техники

**Типы техники**:
- Портальные краны (STS - Ship-to-Shore)
- Козловые краны (RTG - Rubber Tyred Gantry)
- Рельсовые краны (RMG - Rail Mounted Gantry)
- Погрузчики (Reach Stacker)
- Терминальные тягачи
- Автопогрузчики

**Обязательные поля**:
- Идентификатор техники
- Тип и модель
- Грузоподъемность
- Текущий статус (в работе, простой, ремонт)
- Оператор
- Текущее задание
- График обслуживания

### 3. Документооборот и интеграции

#### 3.1 Электронный документооборот (из Solvo.TOS и SeaTerminal)

**Требование**: Автоматизированный обмен данными в форматах EDIFACT, XML, Excel

**Основные документы**:

**Морской транспорт**:
- Коносамент (Bill of Lading)
- Манифест судна
- Карго-план
- Морская декларация
- Агентское извещение

**Железнодорожный транспорт**:
- Железнодорожная накладная
- Акт общей формы
- Коммерческий акт
- Уведомление о прибытии/убытии состава

**Автотранспорт**:
- Товарно-транспортная накладная (ТТН)
- Путевой лист
- Акт приема-передачи груза

**Таможенные документы**:
- Таможенная декларация (ДТ)
- Разрешение на выпуск
- Сертификаты соответствия

**Складские документы**:
- Приходный ордер
- Расходный ордер
- Акт инвентаризации
- Складская опись

#### 3.2 Интеграции с внешними системами

**Обязательные интеграции** (из анализа реальных систем):

1. **ИС РЖД «ЭТРАН»**
   - Автоматический обмен данными о вагонах
   - Получение данных о прибытии составов
   - Передача данных об обработке

2. **Таможенные органы**
   - Электронная подача деклараций
   - Получение разрешений на выпуск
   - Уведомления о досмотрах

3. **Судовые линии и агенты**
   - Получение данных о судозаходах
   - Обмен коносаментами и манифестами
   - Формат EDIFACT: IFTMBC, IFTMBF, IFTMIN

4. **Клиенты и экспедиторы**
   - Web-портал для подачи заявок
   - Отслеживание статуса грузов
   - API для интеграции с клиентскими системами

5. **ERP система предприятия**
   - Передача данных для бухгалтерии
   - Синхронизация справочников
   - Выгрузка актов выполненных работ

### 4. Аналитика и отчетность

#### 4.1 Оперативная отчетность

**Ежедневные отчеты**:
- Отчет о судозаходах (план/факт)
- Отчет о загрузке причалов
- Отчет о приходе/расходе грузов по складам
- Отчет о работе техники
- Отчет об автовизитах

**Реальный пример отчета** (из SeaTerminal):
```
ОТЧЕТ О СУДОЗАХОДАХ НА 18.05.2026

Причал №4:
├─ MSC OSCAR (IMO 9703291)
│  Прибытие: 14:25 (план: 14:00)
│  Обработка: выгрузка 450 / погрузка 380 TEU
│  Прогресс: 234/450 выгружено (52%)
│  Отправление (прогноз): 20:30
│
Причал №2:
├─ VOLGONEFT-251 (IMO 8867234)
│  Прибытие: 08:15
│  Обработка: выгрузка нефтепродуктов 2500т
│  Прогресс: 2100/2500 (84%)
│  Отправление (прогноз): 19:00
```

#### 4.2 Статистическая отчетность

**Ежемесячные отчеты**:
- Грузооборот по типам грузов
- Контейнерооборот (TEU)
- Средняя продолжительность обработки судна
- Коэффициент использования причалов
- Производительность техники
- Анализ простоев
- Финансовые показатели

#### 4.3 KPI и показатели эффективности

**Ключевые метрики** (из исследований McKinsey и данных Solvo.TOS):

1. **Производительность причала**
   - Средняя скорость обработки контейнеров (контейнеров/час)
   - Время стоянки судна (часов)
   - Коэффициент загрузки причалов (%)

2. **Эффективность склада**
   - Оборачиваемость грузов (дней)
   - Коэффициент использования площади (%)
   - Время поиска груза (минут)

3. **Операционная эффективность**
   - Производительность техники (операций/час)
   - Холостой пробег техники (км)
   - Время обработки автовизита (минут)

**Целевые показатели** (из опыта Solvo.TOS):
- Снижение времени стоянки судов: на 20-30%
- Повышение производительности причала: на 25%
- Снижение холостого пробега техники: на 40%
- Снижение времени поиска груза: на 90%

### 5. Пользовательские роли и безопасность

#### 5.1 Роли пользователей

**Роли** (из анализа Solvo.TOS):

1. **Администратор системы**
   - Полный доступ ко всем модулям
   - Управление пользователями
   - Настройка системы

2. **Диспетчер порта**
   - Планирование судозаходов
   - Управление причалами
   - Контроль операций

3. **Диспетчер ж/д фронта**
   - Управление вагонами и составами
   - Планирование ж/д операций

4. **Диспетчер автофронта**
   - Управление автовизитами
   - Выдача пропусков

5. **Кладовщик**
   - Приход/расход грузов
   - Инвентаризация
   - Отчеты по складу

6. **Оператор техники**
   - Получение заданий
   - Подтверждение выполнения операций

7. **Менеджер по работе с клиентами**
   - Просмотр статуса грузов клиента
   - Формирование отчетов для клиентов

8. **Клиент (внешний пользователь)**
   - Подача заявок через web-портал
   - Отслеживание своих грузов
   - Получение документов

#### 5.2 Требования безопасности

**Основные требования**:
- Аутентификация пользователей (логин/пароль, 2FA)
- Разграничение прав доступа по ролям
- Журналирование всех операций
- Шифрование передачи данных (HTTPS, SSL/TLS)
- Резервное копирование данных
- Соответствие требованиям ГОСТ Р 57580.1-2017 (безопасность АСУ ТП)

### 6. Технологические требования

#### 6.1 Архитектура системы

**Требования к архитектуре** (из анализа лучших практик):

1. **Модульная архитектура**
   - Ядро системы (базовые операции)
   - Прикладные модули (расширенный функционал)
   - Возможность подключения новых модулей

2. **Микросервисная архитектура** (для масштабируемости)
   - Сервис управления судами
   - Сервис управления вагонами
   - Сервис управления складами
   - Сервис документооборота
   - Сервис аналитики

3. **API-first подход**
   - RESTful API для всех операций
   - WebSocket для real-time обновлений
   - GraphQL для гибких запросов

#### 6.2 Технологический стек (рекомендации)

**Backend**:
- Node.js + TypeScript (текущий стек) или
- .NET Core (для enterprise-решений) или
- Java Spring Boot (для больших нагрузок)
- База данных: PostgreSQL (текущая) + Redis (кэширование)

**Frontend**:
- React + TypeScript (текущий стек)
- State Management: Redux или MobX
- UI Framework: Material-UI или Ant Design

**Интеграции**:
- Message Queue: RabbitMQ или Apache Kafka
- EDI обработка: библиотеки для EDIFACT
- PDF генерация: puppeteer или PDFKit

**Мониторинг и аналитика**:
- Prometheus + Grafana
- ELK Stack (Elasticsearch, Logstash, Kibana)

#### 6.3 Производительность и масштабируемость

**Требования к производительности**:
- Время отклика API: < 200ms для 95% запросов
- Поддержка 1000+ одновременных пользователей
- Обработка 100+ операций в секунду
- Доступность системы: 99.9% (SLA)

**Масштабируемость**:
- Горизонтальное масштабирование backend
- Репликация базы данных
- Балансировка нагрузки
- Кэширование часто используемых данных
